BACKEND HARDENING BEFORE / AFTER (Task 3-A)

문서 목적

본 문서는 Task 3-A에서 완료된 백엔드 하드닝 작업을 코드 레벨로 설명한다. 인터뷰/코드리뷰에서 바로 확인할 수 있도록 각 변경점마다 Problem, Before, After, Why, Risk/Impact를 포함했다.

  • 범위: AuthController, SecurityConfig, GateWay JwtAuthenticationFilter, AuthService.registerUser, ChatMessageService, WebConfig
  • 참고: Before 코드는 현재 저장소 기준 제거/수정된 동작을 축약 재구성한 스니펫이다(전체 파일 인용 아님).

1) AuthController: /me/password/notoken 제거

Problem

비밀번호 변경 API가 토큰 없는 경로(/me/password/notoken)로도 열려 있으면, 인증 컨텍스트 없이 계정 비밀번호 변경 로직에 도달할 수 있는 설계 취약점이 생긴다.

Before code snippet

java// AuthController (변경 전 패턴)
@PutMapping("/me/password/notoken")
public ResponseEntity<PasswordChangeResponse> changePasswordWithoutToken(
        @RequestBody PasswordChangeRequest request) {
    PasswordChangeResponse response = userService.changePassword(request); // email 기반
    return response.isSuccess() ? ResponseEntity.ok(response)
            : ResponseEntity.badRequest().body(response);
}

After code snippet

java// CoreService/src/main/java/com/example/demo/controller/AuthController.java
@PutMapping("/me/password")
public ResponseEntity<PasswordChangeResponse> changePassword(
        @RequestHeader("Authorization") String token,
        @RequestBody PasswordChangeRequest request) {
    PasswordChangeResponse response = userService.changePasswordByToken(token, request);

    if (!response.isSuccess()) {
        return ResponseEntity.badRequest().body(response);
    }
    return ResponseEntity.ok(response);
}

Why changed

비밀번호 변경을 "인증된 본인" 시나리오로 강제해 계정 탈취/우회 변경 가능성을 줄이기 위함이다.

Risk / Impact

  • 기존 클라이언트가 /me/password/notoken 호출 중이면 즉시 404/정책 차단 영향이 발생한다.
  • 프론트/모바일은 반드시 Authorization: Bearer <JWT> 기반 /api/core/auth/me/password로 마이그레이션해야 한다.

2) SecurityConfig: 비밀번호 변경 라우트 인증 정책 변경

Problem

비밀번호 변경 엔드포인트가 permitAll()에 섞여 있으면, 컨트롤러 코드와 무관하게 보안 필터 체인에서 우선 우회될 수 있다.

Before code snippet

java// SecurityConfig (변경 전 패턴)
.authorizeHttpRequests(auth -> auth
    .requestMatchers(
        "/api/core/auth/signup",
        "/api/core/auth/login",
        "/api/core/auth/me/password",          // 공개로 열려 있던 패턴
        "/api/core/auth/me/password/notoken"   // 공개 변경 경로
    ).permitAll()
)

After code snippet

java// CoreService/src/main/java/com/example/demo/config/SecurityConfig.java
.authorizeHttpRequests(auth -> auth
    .requestMatchers(
        "/api/core/auth/signup",
        "/api/core/auth/login",
        "/api/core/auth/logout"
    ).permitAll()
    .requestMatchers(
        "/api/core/auth/me/password"
    ).authenticated()
)

Why changed

"회원가입/로그인" 같은 공개 경로와 "계정 보안 변경" 경로를 분리해 정책 혼선을 제거하고, 필터 체인 차원에서 최소 권한을 강제하기 위해서다.

Risk / Impact

  • 인증 미첨부 요청은 401/403으로 바뀐다.
  • QA 시나리오에서 비밀번호 변경 테스트는 토큰 발급 절차를 선행해야 한다.

3) GateWay JwtAuthenticationFilter: 공개 경로 축소 + JWT 키 파생 정렬

Problem

1. 공개 경로가 넓으면 Gateway에서 사실상 인증 우회 구간이 커진다.

2. Gateway와 CoreService의 JWT 서명키 파생 방식이 다르면, 한쪽에서 발급한 토큰을 다른 쪽이 불일치로 거부할 수 있다.

Before code snippet

java// JwtAuthenticationFilter (변경 전 패턴)
private static final List<String> PUBLIC_PATHS = Arrays.asList(
    "/api/core/auth/**",
    "/api/core/chat/**",      // 과도하게 넓은 공개 경로
    "/api/core/market/**"
);

private boolean validateToken(String token) {
    SecretKey signingKey = Keys.hmacShaKeyFor(secretKey.getBytes(StandardCharsets.UTF_8));
    Jwts.parserBuilder().setSigningKey(signingKey).build().parseClaimsJws(token);
    return true;
}

After code snippet

java// GateWay/src/main/java/com/example/demo/filter/JwtAuthenticationFilter.java
private static final List<String> PUBLIC_PATHS = Arrays.asList(
        "/api/core/auth/signup",
        "/api/core/auth/login",
        "/api/core/auth/logout",
        "/api/core/auth/oauth2/**",
        "/ws",
        "/ws/**",
        "/topic/**"
);

private boolean validateToken(String token) {
    String base64EncodedSecretKey = Base64.getEncoder()
            .encodeToString(secretKey.getBytes(StandardCharsets.UTF_8));
    SecretKey signingKey = Keys.hmacShaKeyFor(base64EncodedSecretKey.getBytes(StandardCharsets.UTF_8));

    Jwts.parserBuilder()
            .setSigningKey(signingKey)
            .build()
            .parseClaimsJws(token);
    return true;
}

Why changed

  • 공개 경로를 로그인/회원가입/WS handshake 등 필수 범위로 줄여 Gateway 선단에서 먼저 차단하기 위함.
  • JwtTokenProvider와 동일한 키 파생 규칙(Base64 인코딩 후 HMAC key 생성)으로 맞춰 토큰 상호 검증 일관성을 확보하기 위함.

Risk / Impact

  • 기존에 무토큰 접근하던 API가 Gateway에서 즉시 401을 받을 수 있다.
  • 운영 환경에서 jwt.secret/Core JWT secret 설정이 논리적으로 동일하지 않으면 인증 실패가 증가한다.

4) AuthService.registerUser: insert 실패 처리 + catch 롤백 강제

Problem

회원 기본정보 insert 실패(result <= 0) 시 후속 로직이 진행되거나, 예외 catch에서 트랜잭션 롤백이 명시되지 않으면 부분 저장/불일치 데이터 위험이 있다.

Before code snippet

java// AuthService.registerUser (변경 전 패턴)
int result = userMapper.insertUser(user);

// insert 결과 검증 없이 후속 처리 진행 가능
userMapper.initializeUserActivity(request.getEmail(), 50, 0);
hobbyService.registerUserHobbies(request.getEmail(), request.getHobbies());

return SignupResponse.builder().success(true).build();

// catch 에서 롤백 명시 없음

After code snippet

java// CoreService/src/main/java/com/example/demo/service/AuthService.java
int result = userMapper.insertUser(user);

if (result > 0) {
    int initialDopamine = 50;
    int initialPoints = 0;
    userMapper.initializeUserActivity(request.getEmail(), initialDopamine, initialPoints);

    if (request.getHobbies() != null && !request.getHobbies().isEmpty()) {
        hobbyService.registerUserHobbies(request.getEmail(), request.getHobbies());
    }

    return SignupResponse.builder()
            .success(true)
            .email(request.getEmail())
            .initialDopamine(initialDopamine)
            .initialPoints(initialPoints)
            .build();
} else {
    log.error("회원가입 실패 - 사용자 기본 정보 저장 실패, 이메일: {}, insertUser 결과: {}", request.getEmail(), result);
    return SignupResponse.builder().success(false).message("회원가입에 실패하였습니다.").build();
}

// 예외 시 롤백 강제
catch (Exception e) {
    log.error("회원가입 중 오류 발생 - 이메일: {}", request.getEmail(), e);
    TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    return SignupResponse.builder().success(false).message("회원가입 중 오류가 발생했습니다.").build();
}

Why changed

회원가입을 "다중 단계 쓰기" 트랜잭션으로 보고, 첫 단계 실패/중간 예외 모두를 실패로 수렴시켜 데이터 정합성을 지키기 위해서다.

Risk / Impact

  • 이전보다 실패 응답이 명확해져, 클라이언트가 실패 케이스를 더 자주 체감할 수 있다(실제론 정상 동작).
  • DB/매퍼 오류 시 부분 성공이 줄고, 전체 실패로 일관 처리된다.

5) ChatMessageService: 접근 검증 일원화 + productId 신뢰 경로 정리 + 모집완료 규칙 복원 + 이미지 저장 전 인증

5-1) productId를 요청값이 아닌 채팅방 기준으로 파생

Problem

메시지 요청의 productId를 그대로 신뢰하면, 채팅방-상품 불일치 데이터가 저장될 수 있다.

Before code snippet

java// 변경 전 패턴
ChatMessage message = ChatMessage.builder()
        .chatroomId(request.getChatroomId())
        .productId(request.getProductId()) // 클라이언트 입력 신뢰
        .content(request.getContent())
        .build();

After code snippet

java// CoreService/src/main/java/com/example/demo/service/ChatMessageService.java
ChatRoom chatRoom = chatRoomMapper.findChatRoomById(request.getChatroomId(), senderEmail);
validateChatRoomAccess(chatRoom, senderEmail);

ChatMessage message = ChatMessage.builder()
        .chatroomId(request.getChatroomId())
        .senderEmail(senderEmail)
        .productId(chatRoom.getProductId()) // 서버가 채팅방에서 파생
        .content(request.getContent())
        .messageType(request.getMessageType())
        .build();

Why changed

도메인 기준값(chatRoom.productId)을 단일 진실원으로 사용해 위변조/불일치 입력을 차단하기 위해서다.

Risk / Impact

  • 클라이언트가 보내는 productId는 사실상 무시되므로, 클라이언트 필드 의존 로직은 정리 필요.

5-2) validateChatRoomAccess 공통화

Problem

메시지 전송/조회/읽음/이미지 업로드마다 권한 검증이 제각각이면 누락 지점이 생긴다.

Before code snippet

java// 변경 전 패턴: 메서드별 분산 체크
if (chatRoom == null) throw new IllegalArgumentException("존재하지 않는 채팅방입니다.");
if (!userEmail.equals(chatRoom.getSellerEmail()) && !userEmail.equals(chatRoom.getRequestEmail())) {
    throw new IllegalArgumentException("접근 권한이 없습니다.");
}
// 일부 메서드는 위 검증 자체가 누락될 수 있음

After code snippet

java// CoreService/src/main/java/com/example/demo/service/ChatMessageService.java
private void validateChatRoomAccess(ChatRoom chatRoom, String userEmail) {
    if (chatRoom == null) {
        throw new IllegalArgumentException("존재하지 않는 채팅방입니다.");
    }

    if (!isChatParticipant(chatRoom, userEmail)) {
        throw new IllegalArgumentException("해당 채팅방에 접근 권한이 없습니다.");
    }

    Product product = productMapper.findById(chatRoom.getProductId(), chatRoom.getSellerEmail());
    if (product == null) {
        throw new IllegalArgumentException("상품 정보를 찾을 수 없습니다.");
    }

    if (!product.isVisible() && userEmail.equals(chatRoom.getRequestEmail())) {
        ProductRequest productRequest = productRequestMapper.findByProductIdAndRequesterEmail(
                chatRoom.getProductId(), userEmail);

        if (productRequest == null || !"승인".equals(productRequest.getApprovalStatus())) {
            throw new IllegalArgumentException("모집이 완료된 상품은 승인된 사용자만 채팅이 가능합니다.");
        }
    }
}

Why changed

권한 규칙을 단일 메서드로 수렴해 정책 일관성과 유지보수성을 높이기 위해서다.

Risk / Impact

  • 이전에 우연히 통과되던 엣지 케이스가 명시적으로 차단될 수 있다.
  • 예외 메시지 기반 프론트 처리 시 문구 의존이 있다면 확인 필요.

5-3) 모집 완료(비노출) 상품 승인자만 채팅 가능 규칙 복원

Problem

상품이 모집 완료(!product.isVisible()) 상태일 때 승인되지 않은 요청자 접근이 열려 있으면 비정상 대화 접근이 가능해진다.

Before code snippet

java// 변경 전 패턴: 모집완료 상태에 대한 승인검증 누락
if (chatRoom == null || !isParticipant(chatRoom, userEmail)) {
    throw new IllegalArgumentException("접근 불가");
}

After code snippet

java// CoreService/src/main/java/com/example/demo/service/ChatMessageService.java
if (!product.isVisible() && userEmail.equals(chatRoom.getRequestEmail())) {
    ProductRequest productRequest = productRequestMapper.findByProductIdAndRequesterEmail(
            chatRoom.getProductId(), userEmail);

    if (productRequest == null || !"승인".equals(productRequest.getApprovalStatus())) {
        throw new IllegalArgumentException("모집이 완료된 상품은 승인된 사용자만 채팅이 가능합니다.");
    }
}

Why changed

거래 종료/모집 완료 이후 채팅 접근 권한을 비즈니스 규칙(승인 상태)과 맞추기 위해서다.

Risk / Impact

  • 모집 완료 이후 무승인 참여자의 메시지 조회/전송이 실패로 바뀐다(의도된 정책 강화).

5-4) 이미지 업로드 인증 선검증(auth-before-write)

Problem

파일 저장 후 권한을 확인하면, 무권한 요청이 디스크 쓰기를 유발해 저장소 오염 가능성이 있다.

Before code snippet

java// 변경 전 패턴
Path imagePath = chatImageDir.resolve(newFilename);
Files.copy(image.getInputStream(), imagePath, StandardCopyOption.REPLACE_EXISTING);

ChatRoom chatRoom = chatRoomMapper.findChatRoomById(chatroomId, userEmail);
validateChatRoomAccess(chatRoom, userEmail); // write 이후 검증

After code snippet

java// CoreService/src/main/java/com/example/demo/service/ChatMessageService.java
ChatRoom chatRoom = chatRoomMapper.findChatRoomById(chatroomId, userEmail);
validateChatRoomAccess(chatRoom, userEmail); // write 이전 검증

Path imagePath = chatImageDir.resolve(newFilename);
Files.copy(image.getInputStream(), imagePath, StandardCopyOption.REPLACE_EXISTING);

Why changed

인가 실패 요청은 파일시스템에 흔적을 남기지 않도록 하기 위해서다.

Risk / Impact

  • 정책상 무권한 업로드는 더 이른 시점에 실패한다.
  • 저장 실패/검증 실패 경계가 명확해져 로그 해석이 쉬워진다.

6) WebConfig: /chat-images/** 리소스 경로 정렬

Problem

채팅 이미지를 저장하는 실제 경로와 정적 리소스 핸들러 경로가 다르면, 업로드 성공 후 조회 404가 발생한다.

Before code snippet

java// WebConfig (변경 전 패턴)
registry.addResourceHandler("/chat-images/**")
        .addResourceLocations("file:" + System.getProperty("user.dir") + "/src/main/resources/static/chat-images/");

After code snippet

java// CoreService/src/main/java/com/example/demo/config/WebConfig.java
registry.addResourceHandler("/chat-images/**")
        .addResourceLocations("file:" + System.getProperty("user.dir") + "/uploads/chat-images/");

Why changed

ChatMessageService가 실제로 쓰는 경로(user.dir/uploads/chat-images)와 서빙 경로를 맞춰 업로드 즉시 접근 가능하게 하기 위해서다.

Risk / Impact

  • 배포 환경의 user.dir 기준이 달라지면 파일 루트가 달라질 수 있다.
  • 운영 시 워크디렉터리/볼륨 마운트 정책을 함께 고정해야 한다.

검증 상태 및 한계

수행한 확인

  • 코드 단위 정합성 확인: 각 변경점이 현재 구현에 반영되어 있는지 수동 점검.
  • 빌드 검증 시도:
bashcd CoreService
bash ./gradlew compileJava --no-daemon

미해결 검증 한계

현재 실행 환경에는 Gradle toolchain이 요구하는 Java 17이 없어 컴파일 검증이 완료되지 않았다.

핵심 오류:

textCannot find a Java installation ... requirements: {languageVersion=17 ...}
No locally installed toolchains match and toolchain download repositories have not been configured.

즉, 문서 시점의 확인 범위는 "정적 코드 확인"까지이며, 실제 컴파일/런타임 검증은 Java 17 toolchain이 준비된 환경에서 추가 수행이 필요하다.
