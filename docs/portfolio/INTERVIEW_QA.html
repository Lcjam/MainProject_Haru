HARU 인터뷰 Q&A

회원가입/로그인

Q1. 회원가입에서 취미 데이터 검증은 어떻게 했나요?

A. AuthService.registerUser에서 취미 리스트를 순회하면서 categoryId, hobbyId 누락 여부를 먼저 확인하고, HobbyService.isValidCategory, isValidHobby로 존재 검증을 한 뒤 isHobbyInCategory로 관계까지 확인했습니다. 즉, "존재"와 "관계"를 분리 검증했습니다.

Q2. 로그인 보안에서 직접 적용한 포인트는?

A. 로그인 실패 횟수를 DB에 누적하고 5회 이상이면 loginIsLocked를 true로 변경해 계정 잠금을 걸었습니다. 성공 로그인 시 실패 횟수를 0으로 초기화하고 JWT를 발급합니다.

Q3. 로그아웃은 어떻게 처리했나요?

A. 토큰을 단순 삭제하지 않고, 만료 시각까지 블랙리스트에 등록하는 방식으로 무효화했습니다. 이미 발급된 토큰 재사용을 막기 위한 처리입니다.

취미

Q4. 취미 수정 API를 별도로 둔 이유는?

A. 가입 후에도 사용자 취향이 바뀌기 때문에 /api/core/hobbies/user로 조회/수정 엔드포인트를 분리했습니다. 수정 시 기존 사용자 취미를 삭제 후 재등록하도록 구현해 최종 상태를 명확히 맞췄습니다.

Q5. 카테고리-취미 구조를 왜 유지했나요?

A. 프론트에서 "카테고리 먼저 선택 -> 해당 취미 선택" UX를 제공하기 위해서입니다. 백엔드도 같은 구조로 검증해 잘못된 조합 저장을 차단했습니다.

게시판

Q6. 게시글 권한은 어떻게 체크했나요?

A. PostService에서 게시판 존재 여부를 먼저 확인하고, 호스트가 아니면 BoardMember의 활성 상태를 확인합니다. 활성 멤버만 작성/조회 가능하고, 수정은 작성자만, 삭제는 작성자 또는 호스트/ADMIN 멤버로 제한했습니다.

Q7. 게시글 조회 시 추가 처리한 것이 있나요?

A. 상세 조회에서 이미지 목록을 함께 반환하고, 조회수 증가 로직을 같이 수행합니다. 사용자 입장에서 "읽으면 조회수 반영"이 일관되게 동작하도록 했습니다.

채팅

Q8. 채팅을 WebSocket만 쓰지 않고 REST를 같이 쓴 이유는?

A. 실시간 송수신은 WebSocket(STOMP)으로 처리하고, 과거 메시지 조회/읽음 갱신/이미지 메시지 전송은 REST로 처리했습니다. 재접속·초기 진입 시 필요한 데이터 로딩은 REST가 안정적이기 때문입니다.

Q9. WebSocket 인증은 어떻게 했나요?

A. STOMP CONNECT 시 Authorization 헤더를 받아 JWT 검증 후 Principal에 사용자 이메일을 주입했습니다. 이후 @MessageMapping에서 principal.getName()으로 발신자를 식별합니다.

Q10. 채팅방 접근 제어는 어떤 기준인가요?

A. 채팅방 참여자(판매자/요청자)인지 확인하고, 모집 완료 상품의 경우 요청자가 승인 상태일 때만 접근 허용합니다. 승인되지 않으면 메시지로 차단 이유를 반환합니다.

아키텍처/운영

Q11. Gateway를 둔 이유는?

A. 프론트는 /api 단일 진입점만 보도록 만들고, Gateway에서 Core/Assist/FastAPI로 라우팅합니다. 서비스 분리 상태에서도 프론트 호출 구조를 단순하게 유지할 수 있습니다.

Q12. 이번 구현에서 가장 크게 배운 점은?

A. "기능 동작"보다 "경계 조건"이 더 중요하다는 점입니다. 인증, 권한, 상태(승인/잠금/모집완료) 같은 조건을 코드에서 명시적으로 처리해야 실제 서비스에서 오류를 줄일 수 있었습니다.

꼬리질문 대비

Q1. 왜 채팅을 REST + WebSocket으로 병행했나요?

A. 실시간 전송은 @MessageMapping("/chat/send")이 담당하고, 초기 입장 시 과거 메시지 조회/읽음 갱신은 REST API(GET /rooms/{chatroomId}/messages, PUT /rooms/{chatroomId}/messages/read)가 담당합니다. 코드상 역할이 분리되어 있어 재진입 시에도 동작이 안정적입니다.

Q2. 왜 권한 체크를 컨트롤러가 아니라 서비스에서도 했나요?

A. 컨트롤러는 토큰에서 사용자 식별까지만 하고, 실제 도메인 권한(호스트/활성 멤버/작성자)은 PostService에서 검증합니다. 이렇게 해야 같은 서비스 메서드를 다른 경로로 호출해도 동일한 권한 규칙이 유지됩니다.

Q3. 로그인 잠금 정책(5회)의 한계는 무엇인가요?

A. 현재는 실패 횟수 누적 후 잠금까지만 구현되어 있고, 자동 해제나 시간 기반 완화 로직은 코드에 없습니다. 즉, 방어는 가능하지만 운영 편의(잠금 해제 UX)는 별도 정책/기능이 더 필요합니다.

Q4. 취미 수정 시 전체 삭제 후 재등록 방식의 트레이드오프는?

A. 구현이 단순하고 최종 상태를 맞추기 쉽다는 장점이 있지만, 변경량이 적어도 전체를 다시 쓰게 됩니다. 현재 registerUserHobbies는 이 단순성을 선택한 구조입니다.

Q5. 채팅방 접근 시 승인 상태를 왜 추가 검증했나요?

A. ChatService에서 모집 완료 상품일 때 요청자의 승인 상태를 다시 확인합니다. 접근 제어를 채팅 진입 시점에서 확실히 걸어, 프론트 조건 분기만으로는 막기 어려운 우회 접근을 줄이기 위한 선택입니다.
