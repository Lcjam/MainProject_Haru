PAAR Portfolio - Backend Hardening

Intro (작업 범위)

이 문서는 이번 백엔드 하드닝 사이클에서 실제로 수정한 항목만 정리한 PAAR 기록입니다. 범위는 인증 노출면 축소, 회원가입 실패 처리 보강, 채팅 권한/정합성 보정, 채팅 이미지 저장-서빙 경로 일치입니다.

PAAR #1: 인증 노출 제거

Problem

인증 관련 엔드포인트/게이트웨이 예외 경로가 넓으면 불필요한 노출면이 생기고, Core와 Gateway의 JWT 검증 방식이 어긋나면 환경별 인증 실패가 발생할 수 있었습니다.

Action

  • AuthController에서 /me/password/notoken 경로를 제거하고, 비밀번호 변경을 /me/password 흐름으로만 유지했습니다.
  • SecurityConfig에서 비밀번호 변경 라우트(/api/core/auth/me/password)의 인증 정책을 조정했습니다.
  • Gateway JwtAuthenticationFilter의 public path를 축소해 인증 우회 가능 구간을 줄였습니다.
  • Gateway JWT 서명키 파생 로직을 Core 쪽과 맞추는 방향으로 정렬했습니다.

Achievement

비밀번호 변경 경로가 단일 인증 흐름으로 정리됐고, Gateway 레벨에서 공개 경로 범위를 줄여 인증 경계가 더 명확해졌습니다. 또한 JWT 키 파생 기준 정렬로 Core/Gateway 간 토큰 검증 불일치 위험을 낮췄습니다.

Reflection

이번 사이클에서는 "새 보안 기능 추가"보다 기존 경계의 일관성 회복에 집중했습니다. 작은 정책 차이도 실제 인증 실패나 우회 리스크로 이어질 수 있다는 점을 체감했습니다.

코드 근거 경로

  • CoreService/src/main/java/com/example/demo/controller/AuthController.java
  • CoreService/src/main/java/com/example/demo/config/SecurityConfig.java
  • GateWay/src/main/java/com/example/demo/filter/JwtAuthenticationFilter.java

PAAR #2: 회원가입 정합성/트랜잭션 강화

Problem

회원가입에서 insertUser 실패 또는 예외가 발생했을 때, 실패 원인 기록과 트랜잭션 처리 의도가 코드에서 더 명확할 필요가 있었습니다.

Action

  • AuthService.registerUser에서 insertUser 결과가 0 이하일 때 실패 로그를 남기고 실패 응답을 명시적으로 반환하도록 보강했습니다.
  • catch 구간에서 TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()를 적용해 예외 시 롤백 의도를 코드로 고정했습니다.

Achievement

회원가입 실패 케이스가 "침묵 실패"가 아니라 원인 추적 가능한 형태로 정리됐고, 예외 상황의 롤백 처리도 명확해졌습니다.

Reflection

트랜잭션 어노테이션만으로 충분하다고 생각하기 쉬운데, 실제 운영에서는 실패 분기와 로그 맥락까지 같이 남겨야 디버깅 효율이 올라간다는 점을 배웠습니다.

코드 근거 경로

  • CoreService/src/main/java/com/example/demo/service/AuthService.java

PAAR #3: 채팅 권한/데이터 정합성 리팩토링

Problem

채팅 메시지 저장 시 요청 payload를 그대로 신뢰하면 productId 정합성이 깨질 수 있고, API별로 권한 검증 기준이 다르면 닫힌 상품의 요청자 접근 규칙이 흔들릴 수 있었습니다.

Action

  • sendMessage에서 메시지 productId를 request 값이 아니라 chatRoom.getProductId()로 고정했습니다.
  • validateChatRoomAccess 공통 가드를 도입해 채팅방 존재, 참여자 여부, 상품 상태 규칙을 공유 검증으로 묶었습니다.
  • 모집 완료 상품에서 요청자는 승인 상태여야 접근 가능하다는 규칙을 다시 적용했습니다.
  • sendImageMessage에서도 파일 저장 전에 채팅방 조회/권한 검증을 먼저 수행하도록 순서를 보장했습니다.

Achievement

메시지 데이터의 상품 참조가 채팅방 기준으로 일치하게 되었고, 텍스트/이미지/조회/읽음 처리에서 동일한 접근 규칙을 재사용할 수 있게 됐습니다.

Reflection

채팅 하드닝의 핵심은 기능 추가보다 "입력 신뢰 경계"를 어디에 둘지 결정하는 일이라는 점을 명확히 배웠습니다.

코드 근거 경로

  • CoreService/src/main/java/com/example/demo/service/ChatMessageService.java

PAAR #4: 이미지 저장/서빙 경로 안정화

Problem

채팅 이미지 저장 경로와 정적 리소스 서빙 경로가 다르면, 업로드 성공 후 조회 실패가 발생할 수 있었습니다.

Action

  • ChatMessageService의 채팅 이미지 저장 루트를 런타임 디렉토리 uploads/chat-images로 사용하도록 적용했습니다.
  • WebConfig에서 /chat-images/** 리소스 핸들러를 동일한 런타임 경로(file:{user.dir}/uploads/chat-images/)로 매핑해 저장 위치와 서빙 위치를 맞췄습니다.

Achievement

채팅 이미지 업로드 후 접근 경로가 저장 위치와 일치하게 되어 경로 불일치로 인한 조회 실패 가능성을 줄였습니다.

Reflection

파일 업로드 기능은 저장 로직만 맞추는 것이 아니라, 웹 서빙 매핑까지 한 세트로 검증해야 안정화된다는 점을 확인했습니다.

코드 근거 경로

  • CoreService/src/main/java/com/example/demo/service/ChatMessageService.java
  • CoreService/src/main/java/com/example/demo/config/WebConfig.java

Final reflection (배운 점 + 다음 단계)

이번 사이클에서 배운 점은 하드닝 작업의 성격이 "새로운 것 추가"보다 "기존 정책의 경계 통일"에 가깝다는 것입니다. 인증 경로 정리, 실패 분기 명시, 공통 권한 가드, 저장-서빙 경로 일치처럼 작은 정렬 작업이 실제 장애/보안 리스크를 줄이는 데 직접적이었습니다.

다음 단계는 이번에 맞춘 정책을 테스트 코드로 고정해 회귀를 줄이고, Core/Gateway 인증 정책 변경 시 함께 점검하는 체크리스트를 운영 흐름에 넣는 것입니다.
