HARU 포트폴리오 (PAAR)

주니어 개발자 면접에서, 제가 실제로 구현한 기능을 문제 해결 관점으로 설명하기 위해 정리한 문서입니다.

회원가입

Problem

초기 회원가입에서 취미와 카테고리 매핑이 잘못 저장되면, 이후 추천·탐색 기능까지 연쇄적으로 신뢰도가 떨어지는 문제가 있었습니다.

Action

  • AuthService.registerUser에서 이메일, 닉네임, 전화번호 중복을 먼저 검사했습니다.
  • 취미 입력값에 대해 필수값(categoryId, hobbyId) 누락을 확인하고, 유효한 데이터인지 검증 로직을 추가했습니다.
  • isHobbyInCategory 검증으로 카테고리-취미 관계가 맞는 경우만 저장되도록 제한했습니다.
  • 프론트 Signup.tsx와 authAPI.ts에서 백엔드 오류 메시지를 그대로 전달해, 사용자가 어떤 값을 고쳐야 하는지 알 수 있게 했습니다.

Achievement

회원가입 단계에서 잘못된 취미 데이터가 DB에 들어가는 상황을 사전에 막았고, 오류 원인이 명확해져 디버깅과 사용자 대응이 쉬워졌습니다.

Reflection

입력 초기에 도메인 규칙을 강하게 검증하면, 이후 기능에서 예외 처리 코드를 늘리지 않아도 된다는 점을 경험했습니다.

코드 참고: CoreService/src/main/java/com/example/demo/service/AuthService.java, CoreService/src/main/java/com/example/demo/service/HobbyService.java, vite-react-teamsketch/src/pages/account/Signup.tsx, vite-react-teamsketch/src/services/api/authAPI.ts

로그인

Problem

로그인을 단순 성공/실패로만 처리하면 반복 실패 시도에 취약하고, 운영 중 계정 상태를 추적하기 어렵다는 문제가 있었습니다.

Action

  • AuthService.login에서 비밀번호 검증 전에 계정 잠금 상태를 먼저 확인했습니다.
  • 실패 횟수 누적, 5회 이상 잠금, 로그인 성공 시 실패 횟수 초기화를 분리해 처리했습니다.
  • 로그인 시 JWT를 발급하고, 로그아웃 시 블랙리스트 처리로 토큰 재사용을 막았습니다.
  • Login.tsx, authAPI.ts에서 실패 사유를 분기해 사용자에게 상황을 안내하도록 연결했습니다.

Achievement

인증 로직이 단순한 로그인 API를 넘어, 실패 누적과 잠금 상태를 포함한 운영 가능한 인증 흐름으로 정리되었습니다.

Reflection

현재는 잠금 해제 정책이 제한적이어서, 향후에는 시간 기반 자동 해제 같은 운영 정책을 기능으로 확장할 필요가 있습니다.

코드 참고: CoreService/src/main/java/com/example/demo/service/AuthService.java, CoreService/src/main/java/com/example/demo/controller/AuthController.java, vite-react-teamsketch/src/pages/account/Login.tsx, vite-react-teamsketch/src/services/api/authAPI.ts

취미

Problem

사용자 취미는 자주 바뀌는데 수정 결과가 일관되게 반영되지 않으면 마이페이지와 탐색 결과가 실제 사용자 상태와 어긋나는 문제가 생깁니다.

Action

  • HobbyController에서 기준 데이터(카테고리/취미) 조회와 사용자 취미 조회·수정 API를 분리했습니다.
  • HobbyService.registerUserHobbies에서 기존 취미를 정리한 뒤 새 목록을 등록해, 요청 시점의 최종 상태와 DB 상태를 맞췄습니다.
  • 저장 시 registerUserHobby에서 카테고리-취미 관계를 다시 확인해 잘못된 조합 입력을 차단했습니다.

Achievement

사용자 취미 수정 후 재조회 결과가 안정적으로 일치했고, 프론트 입장에서도 상태 동기화 규칙을 단순하게 유지할 수 있었습니다.

Reflection

현재 방식은 구현이 단순한 장점이 있지만 전체 재등록 비용이 있어, 이후에는 변경분만 반영하는 방식으로 개선 여지가 있습니다.

코드 참고: CoreService/src/main/java/com/example/demo/controller/HobbyController.java, CoreService/src/main/java/com/example/demo/service/HobbyService.java, CoreService/src/main/resources/mapper/HobbyMapper.xml

게시판

Problem

게시판은 여러 사용자가 함께 쓰는 기능이라, 권한 검증이 약하면 무권한 조회·수정·삭제 같은 문제가 쉽게 발생합니다.

Action

  • PostService.createPost에서 게시판 존재 여부와 활성 멤버 여부를 먼저 검증했습니다.
  • updatePost는 작성자만 수정 가능하도록 제한했습니다.
  • deletePost는 작성자 또는 호스트/ADMIN만 가능하도록 권한을 분기했습니다.
  • getPostById에서도 멤버십 검증 후 조회수 증가와 응답 생성을 처리해, 접근 정책을 서비스 단에서 일관되게 유지했습니다.

Achievement

게시판 권한 규칙이 컨트롤러가 아니라 서비스 계층에 모여, API가 늘어나도 같은 기준으로 통제할 수 있는 구조를 만들었습니다.

Reflection

권한 거부 사유를 더 세밀하게 분리하면 프론트 UX와 운영 로그 분석에 더 도움이 될 것으로 판단했습니다.

코드 참고: CoreService/src/main/java/com/example/demo/service/PostService.java, CoreService/src/main/java/com/example/demo/controller/board/PostController.java, CoreService/src/main/java/com/example/demo/controller/BoardController.java

채팅

Problem

채팅은 실시간 전송만 구현해서는 부족하고, 접근 권한·히스토리 조회·읽음 처리까지 함께 맞물려야 실제 서비스로 동작합니다.

Action

  • ChatService.createOrGetChatRoom, getChatRoomDetail에서 참여자 및 승인 상태를 검증했습니다.
  • REST는 채팅방 목록/상세/읽음 처리에 사용하고, WebSocket(STOMP)은 실시간 송수신에 집중하도록 역할을 분리했습니다.
  • WebSocketConfig에서 CONNECT 시 JWT를 확인하고 Principal을 주입해, 메시지 발신자 식별을 서버 기준으로 고정했습니다.
  • ChatRoom.tsx, ChatList.tsx, userChatAPI.ts, useChat.ts를 연결해 초기 조회와 실시간 수신이 끊기지 않도록 구성했습니다.

Achievement

재접속이나 초기 진입 상황에서도 채팅 맥락이 유지되었고, 승인되지 않은 사용자 접근을 백엔드에서 일관되게 제한할 수 있었습니다.

Reflection

채팅 품질은 UI보다 경계 조건(권한, 상태, 동기화) 설계에 크게 좌우된다는 점을 배웠고, 이를 서비스 로직에서 명확히 정의하는 습관을 갖게 됐습니다.
