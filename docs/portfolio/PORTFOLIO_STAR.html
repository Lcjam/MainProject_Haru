HARU 포트폴리오 (STAR)

주니어 개발자 면접에서 제가 직접 구현한 기능을 STAR(Situation, Task, Action, Result) 형식으로 설명하기 위한 문서입니다.

1) 회원가입

Situation

취미 기반 서비스에서 회원가입 단계의 데이터가 틀어지면, 이후 추천과 사용자 분류 정확도에 바로 영향을 줬습니다. 특히 취미와 카테고리 매핑이 잘못 저장되는 문제가 가장 우려됐습니다.

Task

중복 계정 생성을 막고, 취미-카테고리 관계가 맞는 데이터만 저장되도록 회원가입 검증 흐름을 백엔드와 프론트에서 끊김 없이 연결하는 것이 목표였습니다.

Action

AuthService.registerUser에서 이메일/닉네임/전화번호 중복을 먼저 확인했습니다. 이후 요청으로 들어온 취미 목록을 순회하며 필수값 누락을 점검하고, HobbyService의 검증 메서드로 카테고리 존재 여부, 취미 존재 여부, 카테고리-취미 관계를 단계적으로 검사했습니다. 검증 통과 시에만 비밀번호 해싱 후 사용자와 취미를 저장하도록 처리했고, 프론트 Signup.tsx와 signupApi를 연결해 실패 사유가 화면에 전달되도록 맞췄습니다.

Result

회원가입 시점에서 잘못된 취미 조합을 조기에 차단해 데이터 일관성을 지켰고, 실패 원인이 명확하게 전달되어 사용자 안내와 개발 중 디버깅이 모두 수월해졌습니다.

코드 참고: CoreService/src/main/java/com/example/demo/service/AuthService.java, CoreService/src/main/java/com/example/demo/service/HobbyService.java, vite-react-teamsketch/src/pages/account/Signup.tsx, vite-react-teamsketch/src/services/api/authAPI.ts

2) 로그인

Situation

초기 로그인은 성공 케이스 중심으로만 보였고, 실패 누적이나 계정 잠금 같은 운영 관점의 방어 로직이 부족했습니다.

Task

로그인 성공/실패/잠금 흐름을 명확히 분리하고, 반복 실패에 대응할 수 있는 정책을 서비스 로직으로 구현하는 것이 필요했습니다.

Action

AuthService.login에서 계정 상태와 잠금 여부를 먼저 검사하도록 순서를 정리했습니다. 비밀번호가 틀리면 실패 횟수를 누적하고 기준 횟수 이상이면 잠금 상태로 전환되게 했습니다. 로그인 성공 시에는 실패 횟수를 초기화하고 JWT를 발급하도록 구성했습니다. 로그아웃은 토큰 블랙리스트 방식으로 무효화했고, 프론트 Login.tsx와 loginApi에서 응답 상태를 화면 처리와 연결했습니다.

Result

무차별 로그인 시도에 대한 기본 방어선을 확보했고, 인증 흐름이 상태별로 분리되어 장애 대응과 원인 파악이 쉬워졌습니다.

코드 참고: CoreService/src/main/java/com/example/demo/service/AuthService.java, CoreService/src/main/java/com/example/demo/controller/AuthController.java, vite-react-teamsketch/src/pages/account/Login.tsx, vite-react-teamsketch/src/services/api/authAPI.ts

3) 취미

Situation

사용자 취향은 가입 이후에도 자주 바뀌기 때문에, 조회만 가능한 구조로는 실제 사용성을 만족시키기 어려웠습니다.

Task

취미/카테고리 조회 API와 사용자 취미 수정 API를 분리해 제공하고, 수정 이후 조회 결과가 일관되게 보장되도록 만드는 것이 과제였습니다.

Action

HobbyController에 전체 취미, 카테고리, 카테고리별 취미, 사용자 취미 조회/수정 엔드포인트를 구성했습니다. HobbyService.registerUserHobbies에서는 기존 사용자 취미를 정리한 뒤 새 요청 목록을 다시 등록하는 방식으로 갱신 로직을 단순화했습니다. 개별 등록 단계에서 카테고리-취미 관계를 다시 검증해 잘못된 조합 저장을 막았습니다.

Result

취미 수정 후 조회 결과가 최종 상태와 맞게 유지되어 데이터 동기화가 안정됐고, 프론트에서 상태 관리할 때 예외 케이스가 줄었습니다.

코드 참고: CoreService/src/main/java/com/example/demo/controller/HobbyController.java, CoreService/src/main/java/com/example/demo/service/HobbyService.java, CoreService/src/main/resources/mapper/HobbyMapper.xml

4) 게시판

Situation

게시판은 여러 사용자가 함께 쓰기 때문에 권한 검증이 약하면 비멤버 접근, 비작성자 수정/삭제 같은 문제가 쉽게 발생할 수 있었습니다.

Task

게시판/게시글 CRUD 전 과정에서 멤버십과 역할(호스트/활성 멤버/작성자/관리자)에 따른 접근 제어를 서비스 계층에 일관되게 반영해야 했습니다.

Action

PostService.createPost에서 게시판 존재를 확인한 뒤, 호스트가 아닌 경우 활성 멤버 여부를 검사했습니다. updatePost는 작성자만 수정 가능하도록 제한했고, deletePost는 작성자 또는 호스트/관리자만 허용했습니다. getPostById는 조회 권한을 멤버 조건으로 제한하면서 상세 조회 시 조회수 증가와 첨부 이미지 조회를 함께 처리했습니다.

Result

게시판 기능이 단순 CRUD를 넘어 권한 정책 중심으로 동작하게 되었고, 권한 관련 예외가 서비스 레이어에서 일관되게 통제됐습니다.

코드 참고: CoreService/src/main/java/com/example/demo/service/PostService.java, CoreService/src/main/java/com/example/demo/controller/board/PostController.java, CoreService/src/main/java/com/example/demo/controller/BoardController.java

5) 채팅

Situation

채팅은 메시지 전송뿐 아니라, 재접속 시 히스토리 확인과 읽음 처리, 참여자 권한 검증이 동시에 맞물리는 기능이라 누락 시 오류가 자주 발생할 수 있었습니다.

Task

WebSocket과 REST 역할을 분리해 안정적으로 운영 가능한 구조를 만들고, 채팅방 접근을 도메인 규칙(참여자/승인 상태)으로 강제하는 것이 목표였습니다.

Action

ChatService.createOrGetChatRoom에서 모집 완료 상태에서는 승인된 요청자만 채팅방에 접근 가능하도록 검증했습니다. getChatRoomDetail에서는 판매자/요청자 참여자 검증 후 읽음 상태를 업데이트하도록 처리했습니다. WebSocketConfig에서 STOMP CONNECT 시 JWT를 검증해 Principal을 주입했고, 프론트 ChatRoom.tsx, ChatList.tsx, userChatAPI.ts, useChat.ts를 통해 실시간 수신과 목록/상세 조회를 연결했습니다.

Result

실시간 송수신과 조회/권한 처리의 책임이 분리되어 유지보수가 쉬워졌고, 승인되지 않은 사용자 접근을 백엔드에서 다시 차단할 수 있게 됐습니다.

코드 참고: CoreService/src/main/java/com/example/demo/service/ChatService.java, CoreService/src/main/java/com/example/demo/controller/ChatController.java, CoreService/src/main/java/com/example/demo/controller/ChatMessageController.java, CoreService/src/main/java/com/example/demo/config/WebSocketConfig.java, vite-react-teamsketch/src/pages/chatroom/ChatRoom.tsx, vite-react-teamsketch/src/pages/chatroom/ChatList.tsx, vite-react-teamsketch/src/services/api/userChatAPI.ts, vite-react-teamsketch/src/services/real-time/useChat.ts
