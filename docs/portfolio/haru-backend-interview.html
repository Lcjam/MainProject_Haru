Haru 백엔드 — 면접 대비 노트

5-서비스 마이크로서비스 팀 프로젝트에서 CoreService 백엔드의 회원가입·로그인·프로필·채팅방·게시판을 담당했습니다. 아래는 내가 직접 구현한 것, 기술적 의사결정의 이유, 예상 질문과 답변 포인트를 정리한 면접 직전 복습용 자료입니다.

⚡ 엘리베이터 피치 (이대로 말하면 됨)

"프론트가 게이트웨이 하나만 바라보는 5-서비스 MSA 구조에서 핵심 도메인 서버인 CoreService를 맡았습니다. 그 안에서 JWT 기반 인증(회원가입·로그인·로그아웃·계정 잠금), 프로필, 그리고 STOMP WebSocket 실시간 채팅과 게시판의 백엔드를 직접 설계·구현했습니다. 특히 채팅은 이력 조회는 REST로, 실시간 송수신은 WebSocket으로 나누고 CONNECT 시점에 JWT를 검증하도록 인증을 직접 붙였습니다. 팀플 종료 후에는 토큰 추출 로직 단일화, 응답 포맷 통일, 단위 테스트 도입 같은 리팩토링으로 코드 품질을 끌어올렸습니다."

1 회원가입 내가 구현

핵심 키워드: 트랜잭션 원자성 + 다단계 검증

내가 구현한 것

  • 이메일 · 닉네임 · 전화번호 중복 검증 후 가입 처리
  • 가입과 동시에 취미(카테고리→취미) 등록 + 초기 활동치(도파민 50, 포인트 0)를 하나의 흐름으로 처리
  • @Transactional + 예외 시 setRollbackOnly() 로 부분 저장 방지
  • 비밀번호는 BCrypt(strength 12) 해싱 (PasswordUtils)

취미 데이터 4단계 검증 (자신 있게 말할 디테일)

  1. categoryId / hobbyId null 체크
  2. isValidCategory() — 존재하는 카테고리인가
  3. isValidHobby() — 존재하는 취미인가
  4. isHobbyInCategory() — 그 취미가 그 카테고리에 실제로 속하는가

회원가입에서 데이터 정합성은 어떻게 보장했나요?

사용자 기본정보 저장, 초기 활동치 설정, 취미 등록을 @Transactional 한 단위로 묶고, 중간에 예외가 나면 setRollbackOnly()로 전체를 롤백했습니다. 취미는 단순 존재 여부뿐 아니라 "취미가 해당 카테고리에 속하는지"까지 검증해 위·변조된 조합을 막았습니다.

"왜 검증을 컨트롤러가 아니라 서비스에 뒀나요?" → 트랜잭션 경계와 비즈니스 규칙은 서비스 책임이고, 컨트롤러는 입력 형식(@Valid)만 본다는 분리 원칙 때문입니다.

CoreService/.../service/AuthService.java (registerUser, @Transactional) · util/PasswordUtils.java

2 로그인 / 로그아웃 내가 구현

핵심 키워드: JWT 발급 + 무차별 대입 방어(5회 잠금) + 로그아웃 블랙리스트

로그인 흐름 (내가 설계한 순서)

이메일 조회 → 계정상태 확인(탈퇴/비활성) → 잠금 확인 → BCrypt 비밀번호 검증 ├ 실패: 실패횟수 +1 → 5회 도달 시 계정 잠금 └ 성공: 실패횟수 0 초기화 + 마지막 로그인 시각 갱신 → JWT 발급 (userId, email, ROLE_*)

로그아웃 = JWT 블랙리스트

  • Stateless JWT는 서버가 강제 만료를 못 시키므로, 로그아웃 토큰을 블랙리스트에 등록해 만료시각까지 차단
  • ConcurrentHashMap 저장 + 1시간마다 만료 토큰 정리 스케줄러

Stateless JWT인데 로그아웃은 어떻게 처리했나요?

JWT는 서명만 맞으면 유효하기 때문에 서버 측에서 무효화가 안 됩니다. 그래서 로그아웃 시 해당 토큰을 만료시각과 함께 블랙리스트에 넣고, 이후 요청에서 블랙리스트 포함 여부를 검사했습니다. 만료된 항목은 스케줄러로 주기적으로 청소합니다.

"인메모리 블랙리스트의 한계는?" → 서버 재시작 시 소실되고 다중 인스턴스 간 공유가 안 됩니다. 그래서 Redis로 옮기는 것을 다음 개선 과제로 잡고 있습니다. (먼저 인정하면 강함)

로그인 보안에서 직접 신경 쓴 점은?

BCrypt strength 12로 해싱했고, 실패 5회 누적 시 계정을 잠가 무차별 대입을 막았으며, 오류 메시지를 "이메일 또는 비밀번호가 잘못됨"으로 통일해 계정 존재 여부가 노출되지 않게 했습니다.

"5회 잠금의 한계는?" → 정상 사용자 락아웃(DoS) 가능성. 해제 정책(시간 경과 후 자동 해제)이나 CAPTCHA 병행이 필요하다고 봅니다.

AuthService.java (login/logout, MAX_FAILED_ATTEMPTS=5) · security/JwtTokenBlacklistService.java · security/JwtTokenProvider.java

4 채팅방 내가 구현 (WebSocket 인증 포함)

⭐ 면접의 메인 무기. 핵심 키워드: REST+STOMP 병행 · CONNECT 시점 JWT 인증 · 권한 기반 승인

왜 REST와 WebSocket을 같이 썼나

REST
  • 채팅방 생성/조회, 목록
  • 이전 메시지 이력 조회·페이징
  • 요청/승인 등 상태 변경
STOMP / WebSocket
  • 실시간 메시지 송수신
  • /topic/room.{id} 구독
  • 위치공유·알림도 같은 채널 체계

WebSocket 인증을 직접 붙인 방법 (내가 구현)

  • HTTP가 아닌 WS 핸드셰이크라 JWT 필터를 못 타므로, ChannelInterceptor.preSend()에서 STOMP CONNECT 프레임의 Authorization 헤더를 직접 검증
  • 검증 성공 시 이메일을 Principal로 세팅 → 이후 메시지 처리에서 사용자 식별
  • 토큰이 없거나 검증 실패면 preSend에서 null 반환 → 연결 거부

"함께하기" 승인 — 권한 제어

  • 채팅방은 상품의 productrequests(함께하기 요청)와 연결
  • 승인 API에서 상품 등록자(seller)가 아니면 403 반환 → 권한 검증을 컨트롤러에서 명시
  • 승인 시 요청 상태 변경 + 모집 인원 증가 + 정원 충족 시 모집 마감 처리

WebSocket 인증은 어떻게 했나요?

STOMP CONNECT 프레임을 가로채는 ChannelInterceptor를 등록해, 연결 시점에 Authorization 헤더의 JWT를 검증했습니다. 성공하면 이메일을 Principal로 세션에 심어 이후 메시지에서 발신자를 식별하고, 실패하면 메시지를 null로 만들어 연결 자체를 끊었습니다.

"매 메시지마다 재검증 안 하나요?" → 현재는 CONNECT 시점 1회 검증입니다. 토큰 만료를 메시지 단에서도 잡으려면 인터셉터에서 SEND 프레임도 검사하도록 확장할 수 있습니다.

채팅방 접근/승인 권한은 어떤 기준인가요?

승인 같은 민감 동작은 상품 등록자만 가능하도록 서버에서 등록자 이메일과 요청자를 대조해 아니면 403을 줍니다. 프론트에서 버튼을 숨기는 것과 별개로 서버에서 한 번 더 막는 게 원칙이라고 봤습니다.

config/WebSocketConfig.java (configureClientInboundChannel/preSend) · controller/ChatController.java (approve, 403) · service/ChatService.java

게시글 권한은 어떻게 체크했나요?

컨트롤러에서 토큰으로 사용자 인증을 먼저 하고, 비즈니스 권한(작성자인지, 멤버인지, 호스트/관리자인지)은 서비스 계층에서 검증했습니다. 예를 들어 수정은 작성자만, 삭제는 작성자·호스트·관리자만 가능하도록 분기했습니다.

"왜 컨트롤러와 서비스 양쪽에서 보나요?" → 컨트롤러는 '누구인지'(인증), 서비스는 '무엇을 할 수 있는지'(인가)로 책임이 다르고, 서비스가 다른 경로로 호출돼도 권한이 보장되게 하려고요.

↻ 팀플 종료 후 개인 리팩토링

시점을 분명히 분리하세요 — "팀플 당시 구현"과 "이후 포트폴리오용 품질 개선"은 둘 다 강점입니다.
