Haru 백엔드 면접 예상 질문 30

신입 백엔드(Java + Spring) · 팀 프로젝트 Haru · 담당: CoreService 인증·채팅·게시판·취미 도메인

모든 답변은 실제 코드베이스로 대조 검증했습니다. 각 항목 하단 코드 근거는 실제 파일 위치입니다. 말할 때는 “문제 → 결정 → 방법 → 결과” 순서로. 모르면 “확인해보겠습니다”가 정답 — 지어내지 마세요.

1. 프로젝트 · 아키텍처

Q1 Haru는 어떤 프로젝트이고, 본인은 무엇을 맡았나요?

취미 공유 · 중고 거래 · 실시간 채팅을 결합한 지역기반 플랫폼입니다. 4인 팀(팀원)에서 저는 CoreService 백엔드의 인증/인가, 실시간 채팅, 게시판, 취미·카테고리 도메인을 담당했습니다.

시스템은 Gateway · CoreService · AssistService로 분리된 마이크로서비스이고, 프론트(React)는 게이트웨이(8080) 한 곳만 바라보게 설계했습니다.

코드 근거 CoreService/src/main/java/com/example/demo/service/{AuthService,ChatService,PostService,HobbyService}.java

Q2 왜 마이크로서비스로 나눴나요? 구성이 어떻게 되나요?

  • GateWay(8080) — 진입점·JWT 인증·라우팅 (Spring Cloud Gateway, WebFlux 기반)
  • CoreService(8081) — 사용자/마켓/보드/채팅/취미 (Spring Boot + MyBatis + Redis + STOMP)
  • AssistService(8082) — OCR/SMS/번역/AI 챗봇 보조 등 부가 기능

핵심 도메인(Core)과 외부 연동·부가기능(Assist)을 분리해, 보조 기능의 장애나 외부 API 이슈가 핵심 서비스로 번지지 않게 한 것이 목적입니다. 프론트가 게이트웨이만 보므로 내부 서비스 URL 변경이 프론트에 영향을 주지 않습니다.

코드 근거 GateWay/src/main/java/com/example/demo/config/RouteConfig.java · CLAUDE.md 서비스/포트 표

Q3 게이트웨이에서 JWT를 검증하는데 CoreService에서 또 검증합니다. 중복 아닌가요?

의도된 이중 방어(defense in depth)입니다. 게이트웨이는 유효하지 않은 토큰을 가장 앞단에서 401로 차단해 내부 서비스로 트래픽이 새지 않게 하고, CoreService는 게이트웨이를 우회한 직접 호출도 막기 위해 스스로 다시 검증해 SecurityContext를 구성합니다.

두 서비스는 동일한 시크릿(JWT_SECRET 환경변수)을 공유하도록 설정했습니다. 게이트웨이는 통과 여부만 판단하고, 사용자 신원 확정(권한 주입)은 각 서비스가 책임집니다.

코드 근거 GateWay/.../filter/JwtAuthenticationFilter.java · CoreService/.../security/JwtAuthenticationFilter.java · application.properties jwt.secret=${JWT_SECRET}

Q4 인증이 필요 없는 요청은 어떻게 구분하나요?

공개 경로를 게이트웨이와 CoreService 양쪽에서 일치시켜 관리합니다. 회원가입/로그인/로그아웃, 회원가입 화면에서 필요한 취미·카테고리 조회, WebSocket 핸드셰이크(/ws, /topic/**) 등이 공개입니다. 게이트웨이는 PUBLIC_PATHS 목록으로, Core는 SecurityConfig의 permitAll()로 화이트리스트를 관리합니다.

코드 근거 GateWay JwtAuthenticationFilter PUBLIC_PATHS · CoreService/.../config/SecurityConfig.java authorizeHttpRequests

2. 인증 / 인가 (Spring Security + JWT)

Q5 로그인부터 인증까지 전체 흐름을 설명해 주세요.

  1. 로그인 요청 → 이메일로 사용자 조회, 계정 상태 → 잠금 여부 → 비밀번호 순으로 검사
  2. 성공 시 실패 카운트 초기화, JwtTokenProvider.createToken()으로 JWT 발급(subject=email, userId·roles 클레임)
  3. 이후 요청은 Authorization: Bearer <token> 헤더로 전달
  4. JwtAuthenticationFilter(OncePerRequestFilter)가 토큰 검증 + 블랙리스트 확인 후 SecurityContext에 인증 주입
코드 근거 AuthService.login() · JwtTokenProvider.createToken()/getAuthentication() · security/JwtAuthenticationFilter.doFilterInternal()

Q6 발급하는 JWT의 구조와 서명 방식은?

HS256(HMAC-SHA256) 대칭키 서명을 사용합니다. 페이로드에는 sub(email), userId, roles, iat, exp를 담고, 만료는 24시간(86400000ms, 설정값)입니다. 서명 검증은 파싱 시 서명·만료를 함께 확인합니다.

대칭키를 택한 이유는 토큰을 발급·검증하는 주체가 우리 서비스 내부(Gateway·Core)로 한정되기 때문입니다. 외부에 검증을 위임해야 한다면 비대칭(RS256)이 더 적합하다는 점도 알고 있습니다.

코드 근거 JwtTokenProvider (SignatureAlgorithm.HS256) · application.properties jwt.expiration=86400000

Q7 세션 대신 JWT를 쓴 이유는?

마이크로서비스라 서버가 상태를 들고 있으면 서비스마다 세션 공유가 필요합니다. JWT는 토큰 자체에 신원이 담겨 Stateless하게 검증할 수 있어, 게이트웨이·Core가 각자 독립 검증하기에 적합했습니다. Spring Security도 SessionCreationPolicy.STATELESS로 설정했습니다.

대가로 발급된 토큰을 서버가 강제 무효화하기 어렵다는 단점이 있어, 로그아웃을 위해 블랙리스트를 별도로 뒀습니다(Q8).

코드 근거 SecurityConfig sessionCreationPolicy(STATELESS)

Q8 Stateless JWT인데 로그아웃은 어떻게 구현했나요?

로그아웃 시 해당 토큰을 블랙리스트에 등록하고, 이후 인증 필터가 블랙리스트에 있으면 만료 전이라도 401로 거부합니다. 블랙리스트 항목은 토큰의 원래 만료 시각까지만 보관하고 주기적으로 정리합니다(만료 후엔 어차피 검증에서 걸리므로 보관 불필요).

코드 근거 AuthService.logout() → JwtTokenBlacklistService.addToBlacklist() · JwtAuthenticationFilter의 isBlacklisted 분기

Q9 그 블랙리스트는 어디에 저장되나요? 한계와 개선점은? 심화

솔직히 현재는 애플리케이션 인메모리(ConcurrentHashMap)에 두고 1시간마다 만료분을 정리합니다. 한계는 명확합니다 — ① 서버 재시작 시 유실, ② 인스턴스가 여러 대면 공유되지 않음.

개선안은 이미 프로젝트에서 쓰고 있는 Redis로 옮기는 것입니다. SET blacklist:<jti> 1 EX <남은초> 형태로 저장하면 TTL로 자동 만료되고 인스턴스 간에도 공유됩니다. 여기까지 설계는 했고, 실제 이관은 후속 과제로 남겨둔 상태입니다.

코드 근거 JwtTokenBlacklistService ConcurrentHashMap + ScheduledExecutorService 1시간 cleanup / Redis는 RedisConfig에 이미 구성됨

Q10 비밀번호는 어떻게 저장하나요?

BCrypt(strength 12)로 해싱해 저장합니다. BCrypt는 솔트를 해시 문자열에 자체 포함하므로 별도 솔트 컬럼은 레거시 호환용으로만 남아 있고 검증엔 쓰지 않습니다. 평문은 절대 저장하지 않고, 검증은 matches(input, storedHash)로 합니다.

코드 근거 util/PasswordUtils.java new BCryptPasswordEncoder(12), encode()/matches()

Q11 무차별 대입(brute-force)에는 어떻게 대응했나요?

로그인 상태를 성공 / 실패 / 잠금으로 분리했습니다. 비밀번호가 틀리면 실패 횟수를 누적하고, 5회(MAX_FAILED_ATTEMPTS) 이상이면 계정을 잠금 처리합니다. 로그인 성공 시 실패 카운트를 0으로 초기화합니다. 잠긴 계정은 비밀번호 검사 이전에 먼저 걸러집니다.

코드 근거 AuthService.login() MAX_FAILED_ATTEMPTS=5, updateFailedLoginAttempts, updateLoginLockStatus

Q12 Spring Security 필터 체인은 어떻게 구성했나요?

  • csrf().disable() — 토큰 기반 무상태 API라 CSRF 세션 쿠키를 쓰지 않음
  • SessionCreationPolicy.STATELESS
  • 커스텀 JwtAuthenticationFilter를 UsernamePasswordAuthenticationFilter 앞에 배치
  • 인증 실패는 JwtAuthenticationEntryPoint로 일관된 401 응답
  • 공개 경로 permitAll(), 관리자 경로 hasRole("ADMIN"), 나머지 authenticated()
코드 근거 config/SecurityConfig.java filterChain()

3. 실시간 채팅 (STOMP + Redis Pub/Sub)

Q13 실시간 채팅 아키텍처를 설명해 주세요. 핵심

WebSocket 위에 STOMP를 얹고, 서버 간 확산은 Redis Pub/Sub로 처리합니다. 메시지 한 통의 흐름:

  1. 클라이언트가 /app으로 전송 → ChatMessageService.sendMessage()
  2. DB에 메시지 저장 + 채팅방 마지막 메시지 갱신
  3. 메시지를 Redis 채널(chat)에 발행 (convertAndSend)
  4. RedisMessageListener.onMessage()가 구독 → /topic/chat.{roomId}로 STOMP 브로드캐스트
  5. 해당 방을 구독 중인 클라이언트가 실시간 수신
코드 근거 ChatMessageService.sendMessage() · listener/RedisMessageListener.onMessage() · config/RedisConfig.java · config/WebSocketConfig.java

Q14 왜 굳이 Redis Pub/Sub를 끼웠나요? STOMP만으로 안 되나요? 심화

메시지 브로커로 스프링 내장 SimpleBroker(인메모리)를 씁니다. 이건 단일 인스턴스 안에서만 구독자에게 전달되므로, 서버를 여러 대로 확장하면 A서버에 붙은 사용자와 B서버에 붙은 사용자가 서로 메시지를 못 받습니다.

그래서 발행(publish)과 브로드캐스트(broadcast)를 분리했습니다. 어느 서버가 메시지를 받든 Redis 채널에 발행하면 모든 서버의 구독 리스너가 받아 각자의 STOMP 구독자에게 전달합니다. 인메모리 브로커의 수평 확장 한계를 Redis로 메운 구조입니다.

코드 근거 WebSocketConfig enableSimpleBroker("/topic") · RedisConfig RedisMessageListenerContainer + ChannelTopic("chat")

Q15 WebSocket은 HTTP 필터를 안 타는데 인증은 어떻게 했나요? 심화

맞습니다. 그래서 HTTP JwtAuthenticationFilter는 WebSocket 업그레이드 요청을 제외(shouldNotFilter)하고, 인증은 STOMP 채널 인터셉터에서 합니다. configureClientInboundChannel의 ChannelInterceptor.preSend에서 STOMP CONNECT 프레임일 때 Authorization 헤더의 JWT를 검증하고, 통과하면 이메일을 담은 Principal을 세션에 주입합니다. 실패하면 null을 반환해 연결 자체를 거부합니다.

코드 근거 WebSocketConfig configureClientInboundChannel (StompCommand.CONNECT → validateToken → accessor.setUser) · security/JwtAuthenticationFilter.shouldNotFilter

Q16 STOMP 목적지 경로 규약은 어떻게 정했나요?

  • /app/** — 클라이언트 → 서버 (application destination prefix)
  • /topic/** — 서버 → 구독자 (SimpleBroker)
  • 채팅방별 브로드캐스트는 /topic/chat.{chatroomId}로 방 단위 구독
  • 하트비트 10초/10초로 유휴 연결 감지
코드 근거 WebSocketConfig setApplicationDestinationPrefixes("/app"), setHeartbeatValue · RedisMessageListener destination

Q17 아무나 채팅방에 들어올 수 있나요? 접근 권한은?

도메인 규칙으로 막습니다. 상품이 모집 완료(비공개) 상태면, 판매자이거나 요청이 “승인”된 사용자만 채팅방에 접근할 수 있습니다. 승인되지 않은 사용자는 서비스 계층에서 차단합니다. 채팅방 상세 조회 시에도 판매자/요청자 등 참여자인지 다시 검증합니다.

코드 근거 ChatService.createOrGetChatRoom()/getChatRoomDetail() !"승인".equals(productRequest.getApprovalStatus()), 참여자 검증

Q18 읽음 처리와 안 읽은 메시지 수는 어떻게 하나요?

채팅방 상세에 들어오면 해당 사용자의 안 읽은 메시지를 읽음 처리하고, 목록 화면에서는 방마다 countUnreadMessages(roomId, userEmail)로 안 읽은 개수를 계산해 노출합니다. 읽음 상태는 메시지 단위로 관리합니다.

코드 근거 ChatService(getChatRoomDetail 읽음 갱신) · chatMessageMapper.countUnreadMessages

Q19 메시지가 유실되거나 순서가 꼬일 수 있지 않나요? 심화

전송은 DB 저장 → Redis 발행 순서라, 저장된 메시지는 발행이 실패해도 남습니다(발행 실패는 로깅 후 무시). 재접속 시 REST 히스토리 조회로 복구되므로 “저장됐는데 못 받는” 상황은 화면 갱신으로 회복됩니다.

다만 개선할 지점을 알고 있습니다: sendMessage()가 @Transactional이라 커밋 전에 Redis 발행이 일어납니다. 이론상 이후 롤백되면 “실제로 없는 메시지”가 브로드캐스트될 수 있습니다. 트랜잭션 커밋 이후 발행(afterCommit)으로 바꾸는 것이 더 안전하다고 봅니다.

코드 근거 ChatMessageService.sendMessage() @Transactional → saveChatMessage → redisTemplate.convertAndSend (try/catch)

4. 게시판 · 취미

Q20 게시판 권한 모델을 설명해 주세요.

단순 CRUD가 아니라 역할 기반 접근제어를 서비스 계층에 뒀습니다.

  • 글 작성: 게시판 호스트이거나 ACTIVE 멤버만
  • 수정: 작성자 본인만
  • 삭제: 작성자 · 호스트 · 게시판 ADMIN 멤버만
  • 조회: 게시판 멤버(호스트/ACTIVE)만
코드 근거 PostService.createPost()/updatePost()/deletePost()/getPostById() — hostEmail·BoardMember status/role 검사

Q21 상세 조회 시 조회수 증가는 어떻게 처리했나요?

getPostById()는 @Transactional 안에서 멤버 권한 검증 → 이미지 목록 조회 → 조회수 증가를 함께 처리합니다. 하나의 트랜잭션이라 조회 응답과 조회수 반영이 어긋나지 않습니다. (대량 트래픽이라면 조회수는 별도 비동기/버퍼링이 낫다는 점도 인지하고 있습니다.)

코드 근거 PostService.getPostById() increaseViewCount(postId)

Q22 회원가입 때 취미 데이터는 어떻게 검증하나요?

취미는 반드시 카테고리와 함께 들어오는데, “이 취미가 정말 그 카테고리에 속하는가”를 서비스에서 검증합니다. isHobbyInCategory(hobbyId, categoryId)가 false면 IllegalArgumentException으로 저장을 막습니다. 잘못된 취미–카테고리 조합이 저장되면 이후 추천·분류 정확도가 깨지기 때문에, 입력 시점에서 데이터 정합성을 지키는 것이 목적입니다.

코드 근거 HobbyService.registerUserHobby() hobbyMapper.isHobbyInCategory

Q23 취미 수정은 어떻게 구현했나요? 그 방식의 트레이드오프는? 심화

registerUserHobbies()에서 기존 취미를 전부 삭제하고 새 목록으로 재등록하는 방식입니다. diff(추가/삭제 계산) 로직이 필요 없어 구현이 단순하고, 수정 후 조회 결과가 항상 최종 상태와 일치합니다. 전체가 @Transactional이라 중간 실패 시 원자적으로 롤백됩니다.

단점은 변경이 없어도 매번 전량 삭제·삽입이라는 것 — 취미 개수가 적어 문제되지 않지만, 대량이라면 delta 방식이 효율적입니다.

코드 근거 HobbyService.registerUserHobbies() deleteAllUserHobbies 후 반복 registerUserHobby

5. DB · 트랜잭션 · 데이터

Q24 @Transactional은 어디에, 왜 붙였나요?

여러 테이블 쓰기가 한 흐름으로 묶이는 서비스 메서드에 붙였습니다 — 회원가입(사용자+활동+취미), 로그인(실패카운트/잠금/시간), 채팅 전송(메시지+채팅방 갱신), 게시글 삭제(글+이미지), 취미 재등록 등. 목적은 부분 저장 방지(원자성)입니다. 예를 들어 사용자만 저장되고 취미가 안 들어가는 어중간한 상태를 막습니다.

코드 근거 AuthService·ChatService·PostService·HobbyService·ChatMessageService 각 @Transactional

Q25 회원가입 중간에 예외가 나면 어떻게 되나요?

@Transactional이므로 기본적으로 런타임 예외에 롤백됩니다. 추가로 try/catch로 예외를 잡아 사용자에게 실패 메시지를 돌려주면서도, TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()를 호출해 잡은 예외라도 반드시 롤백되도록 명시했습니다. “예외를 catch하면 롤백이 안 될 수 있다”는 함정을 피한 부분입니다.

코드 근거 AuthService.registerUser() catch 블록 setRollbackOnly()

Q26 ORM(JPA)이 아니라 MyBatis를 쓴 이유는?

팀 합의로 MyBatis를 사용했습니다. 채팅방 목록의 안 읽은 개수 집계, 취미–카테고리 조인 검증처럼 SQL을 직접 제어하고 눈으로 확인하고 싶은 쿼리가 많아, XML 매퍼로 다루는 편이 팀에 명확했습니다. 대신 테이블 케이싱 혼재(Users vs boards) 같은 이슈는 코드/매퍼 레벨 정합으로 맞췄습니다. JPA의 영속성 컨텍스트·연관관계 매핑 장점도 이해하고 있고, 도메인이 정형화되면 JPA가 생산성에 유리하다고 봅니다.

코드 근거 CoreService/src/main/resources/mapper/*.xml (HobbyMapper 등) · CLAUDE.md 케이싱 정합 규약

Q27 게시글 삭제는 물리 삭제인가요?

소프트 삭제입니다. 실제 파일이나 레코드를 지우지 않고 삭제 상태로 표시합니다. 실수 복구·이력 추적에 유리하고, 채팅/게시판처럼 참조가 얽힌 데이터에서 연쇄 삭제로 인한 참조 깨짐을 피하기 위함입니다.

코드 근거 PostService.deletePost() 주석 “소프트 삭제”, postMapper.deletePost(postId, email)

6. 트러블슈팅 · 성장

Q28 프로젝트에서 가장 까다로웠던 문제와 해결은? 핵심

WebSocket 인증이 가장 헷갈렸습니다. 처음엔 HTTP JWT 필터가 당연히 WebSocket에도 적용될 줄 알았는데, 핸드셰이크 이후 STOMP 프레임은 그 필터를 타지 않아 인증이 비어 있었습니다. 원인을 파고들어, HTTP 필터에서는 WebSocket 업그레이드 요청을 제외하고 STOMP CONNECT 시점에 채널 인터셉터로 JWT를 검증해 Principal을 주입하도록 분리해 해결했습니다. “인증을 어느 계층에서 하느냐”를 프로토콜 특성에 맞춰 다시 설계한 경험입니다.

(부가) 채팅방 목록이 특정 상황에서 500이 났는데, productrequests 테이블 누락이 원인이었습니다. 매퍼들이 이 테이블을 참조해 없으면 MySQL 1146→500이 나는 것을 로그로 추적해 스키마를 복구했습니다.

코드 근거 WebSocketConfig 채널 인터셉터 · JwtAuthenticationFilter.shouldNotFilter · ChatService(productrequests 참조)

Q29 코드 품질을 위해 신경 쓴 부분이 있나요?

  • 토큰 추출 로직 단일화: 여기저기 흩어진 Bearer 제거 코드를 TokenUtils.extractTokenWithoutBearer() 한 곳으로 통합
  • 로깅 표준화: System.out.println 대신 SLF4J logger 사용
  • 테스트 도입: 토큰 유틸부터 첫 단위 테스트를 작성해 회귀 안전망 확보
  • 시크릿 외부화: JWT 시크릿·DB 비밀번호를 하드코딩하지 않고 .env/환경변수로 주입

“동작만 하는 코드”에서 멈추지 않고 중복 제거·설정 외부화·테스트로 다듬는 습관을 들이려 했습니다.

코드 근거 util/TokenUtils.java · application.properties ${JWT_SECRET}/${DB_PASSWORD} · 단위 테스트(토큰 유틸)

Q30 다시 한다면 개선하고 싶은 점은?

  • 토큰 블랙리스트를 Redis(TTL)로 이관 — 재시작 유실·다중 인스턴스 공유 문제 해결 (Q9)

완벽하진 않았지만 한계를 인지하고 개선 우선순위를 말할 수 있다는 점이 지금 제 강점이라고 생각합니다.

7. 압박 꼬리질문 (심화 · 약점 파고들기)

기본 답변 뒤에 면접관이 “그러면 이건요?”로 파고드는 질문들입니다. 핵심은 방어가 아니라 인지 — 한계를 인정하고 개선 방향을 구체적으로 말하면 오히려 가점입니다. 아래 일부는 실제 코드의 미구현 갭이라 솔직하게 답하도록 썼습니다.

P1 블랙리스트를 매 요청마다 조회하면, 결국 서버가 상태를 보는 거잖아요. JWT의 stateless 장점이 사라지는 것 아닌가요?

맞는 지적입니다. 블랙리스트는 순수 stateless를 일부 포기하는 대신 즉시 로그아웃(강제 무효화)을 얻는 트레이드오프입니다. 다만 조회는 토큰당 단순 존재 확인 한 번이고 DB 왕복이 아니라 부담이 작습니다.

더 정석적인 대안도 압니다 — 액세스 토큰을 짧게(예: 15분) + 리프레시 토큰 구조로 가면, 블랙리스트 없이도 “곧 만료되니 무효화 지연이 짧다”로 해결됩니다. 이 프로젝트는 단일 만료 24h 토큰이라 즉시 로그아웃을 위해 블랙리스트를 택한 것이고, 리프레시 토큰 도입이 다음 개선지점입니다.

코드 근거 JwtTokenBlacklistService.isBlacklisted (containsKey) · jwt.expiration=86400000(단일 만료)

P2 게이트웨이와 Core가 같은 대칭키(HS256)를 나눠 갖는데, 시크릿이 유출되면요? 서비스가 10개로 늘면 시크릿 배포는?

대칭키라 유출 시 서명 위조가 가능하고, 공유 주체가 늘수록 노출면이 커지는 게 맞습니다. 지금은 하드코딩을 없애고 JWT_SECRET 환경변수로만 주입해 최소 방어는 했습니다.

규모가 커지면 두 방향을 고려합니다 — ① 비대칭(RS256)으로 바꿔 발급자만 개인키, 나머지는 공개키로 검증하면 검증 측에 비밀을 안 뿌려도 됩니다. ② 시크릿을 Vault·AWS Secrets Manager 같은 중앙 관리로 배포·로테이션. 유출 대응으로는 시크릿 교체 + 전 토큰 무효화(만료 단축)입니다.

코드 근거 application.properties jwt.secret=${JWT_SECRET:...} — Gateway와 동일해야 함(주석 명시)

P3 WebSocket은 CONNECT 때 한 번만 인증하죠. 소켓이 몇 시간 유지되는 동안 토큰이 만료되거나 로그아웃하면, 그 연결은 계속 살아있지 않나요? 실제 갭

정확합니다. 현재 구조는 연결 시점 1회 검증이라, 연결 이후 토큰 만료·로그아웃(블랙리스트 등록)이 기존 소켓에는 반영되지 않습니다. 인정하는 한계입니다.

개선안: ① 메시지 전송(SEND) 시점에도 인터셉터에서 만료·블랙리스트를 재검증, ② 토큰 만료 시각에 맞춰 서버가 소켓을 강제 종료, ③ 짧은 액세스 토큰 + 재연결. 최소 비용으로는 ①이 현실적입니다.

코드 근거 WebSocketConfig.preSend — if (StompCommand.CONNECT.equals(...)) 에서만 검증

P4 Redis가 죽으면 채팅은 어떻게 되나요?

발행이 try/catch로 감싸져 있어 서버가 죽진 않고, 메시지는 DB에 이미 저장됩니다. 다만 실시간 브로드캐스트만 실패해 상대는 그 순간 즉시 수신을 못 하고, 새로고침/재접속 시 REST 히스토리로 확인하게 됩니다. 즉 Redis는 실시간 전달의 단일 장애점(SPOF)입니다.

개선: Redis Sentinel/Cluster로 이중화, 발행 실패 시 로컬 브로커로 폴백(같은 인스턴스 구독자라도 전달), 그리고 클라이언트 재연결 시 미수신 구간 REST 동기화를 명확히 합니다.

코드 근거 ChatMessageService.sendMessage — saveChatMessage(선) → convertAndSend try/catch(후)

P5 Redis Pub/Sub는 fire-and-forget이라, 구독자가 순간 끊겨 있으면 그 사이 메시지는 못 받죠. 유실 아닌가요?

Pub/Sub 자체는 저장·재생이 없어 그 순간 구독자가 없으면 그 방송은 사라지는 게 맞습니다. 하지만 우리 설계에서 “원본 메시지”는 DB에 영속되므로, 구독 리스너가 놓친 것은 실시간 화면 갱신일 뿐 데이터 유실은 아닙니다. 재접속 시 REST로 메꿉니다.

“무조건 실시간 전달 보장”이 요구된다면 Pub/Sub 대신 Redis Streams / 메시지 큐(Kafka 등)로 consumer offset·재처리를 두는 게 맞습니다. 채팅 규모 대비 과하다고 판단해 현재 구조를 택했습니다.

코드 근거 RedisConfig ChannelTopic 기반 Pub/Sub(비영속) + chat_messages 테이블(영속)

P6 5회 실패로 잠그면, 공격자가 남의 계정을 일부러 5번 틀려서 잠가버리는 “계정 잠금 DoS”가 되잖아요? 그리고 한 번 잠기면 어떻게 풉니까? 실제 갭

둘 다 유효한 지적이고, 솔직히 현재 구현의 약점입니다. ① 영구 잠금 방식이라 표적 계정을 잠가버리는 남용이 가능하고, ② 코드상 자동/시간 기반 해제 경로가 없어 지금은 관리자·DB 개입이 필요합니다.

개선안이 분명합니다 — 영구 잠금 대신 시간 기반 잠금(예: 15분 쿨다운 후 자동 해제), IP·디바이스 단위 레이트리밋, 실패 누적 시 CAPTCHA 병행. 이렇게 하면 정당 사용자를 오래 막지 않으면서 무차별 대입 속도를 떨어뜨립니다.

코드 근거 AuthService — updateLoginLockStatus(email,true)만 존재, 로그인 흐름에 unlock 없음 · 잠금해제 컬럼(시간) 미구현

P7 취미 목록을 줄 때 취미마다 카테고리를 따로 조회하던데, 그거 N+1 아닌가요? 실제 갭

맞습니다. getAllHobbiesWithCategories()가 취미 목록을 가져온 뒤 취미 하나당 getCategoriesByHobbyId를 반복 호출해 전형적인 N+1입니다. 취미 수가 적어 지금은 체감 문제가 없지만 구조적으로 비효율이 맞습니다.

해결: 취미–카테고리를 조인한 단일 쿼리로 가져와 MyBatis <collection> 결과 매핑(1:N)으로 묶거나, 카테고리를 한 번에 조회해 메모리에서 그룹핑하면 쿼리 1~2회로 끝납니다.

코드 근거 HobbyService.getAllHobbiesWithCategories() — for (Hobby h : hobbies) getCategoriesByHobbyId(h.getHobbyId())

P8 상세 조회마다 조회수 UPDATE가 나가면, 인기 글은 같은 행에 쓰기가 몰려 경합이 심하지 않나요?

맞습니다. 조회 = 쓰기가 되어 인기 글의 단일 행에 UPDATE 락 경합·쓰기 증폭이 생깁니다. 프로젝트 트래픽에선 괜찮지만 확장 시 병목이 될 수 있습니다.

개선: Redis에서 INCR로 카운트하고 주기적으로 DB에 배치 반영, 또는 조회 이벤트를 비동기 큐로 흘려 집계. 실시간 정확도가 덜 중요한 지표라 eventual consistency로 충분합니다.

코드 근거 PostService.getPostById() 내부 postMapper.increaseViewCount(postId) (조회 트랜잭션에 포함)

P9 소프트 삭제면, 삭제된 글이 목록·검색·상세에 안 섞이게 하려면 모든 조회 쿼리에 조건이 붙어야 하죠. 하나라도 빠지면 삭제된 게 노출되는데요?

정확한 위험입니다. 소프트 삭제의 대가는 모든 읽기 경로가 삭제 플래그를 필터해야 한다는 것이고, 한 군데라도 누락되면 유령 데이터가 노출됩니다.

완화책: ① 매퍼 쿼리 컨벤션 통일(조회는 항상 WHERE is_deleted = 0), ② 공통 조건을 기본 조회 SQL/뷰로 캡슐화, ③ “삭제 포함 조회”는 별도 명시적 메서드로만. 규칙을 코드가 아니라 구조로 강제하는 방향이 안전합니다.

코드 근거 PostService.deletePost() “소프트 삭제” · 조회 쿼리들이 삭제 상태를 필터해야 하는 구조

P10 취미 수정이 “전부 삭제 후 재삽입”이면, 같은 사용자가 두 요청을 동시에 보내면 한쪽이 다른 쪽을 날려버리지 않나요?

이론상 그렇습니다. 각 요청이 @Transactional로 원자적이긴 해도 서로 겹치면 마지막 커밋이 이깁니다(last-writer-wins). 다만 이 동작은 본인이 자기 취미를 수정하는 시나리오라 동시 수정 가능성이 매우 낮아 의도적으로 단순화한 선택입니다.

정말 필요하면 낙관적 락(version 컬럼)으로 충돌을 감지하거나, 전량 교체 대신 delta(추가/삭제) 반영으로 경합면을 줄일 수 있습니다.

코드 근거 HobbyService.registerUserHobbies() — deleteAllUserHobbies 후 재삽입(단일 사용자 self-edit 전제)

8. 이력서 ↔ 질문 매핑 치트시트

면접관은 이력서 한 줄을 읽고 질문합니다. 각 줄이 어떤 질문을 부르는지(Q=기본, P=압박), 30초 방어 요지, 그리고 밟으면 위험한 지뢰를 매핑했습니다. 노란색은 “먼저 인정하고 개선안을 말할” 자백 포인트.

✅ 먼저 꺼내라 — 선제 자백 3 (자기객관화 = 가점)

  1. 토큰 블랙리스트가 인메모리라 재시작 유실·다중 인스턴스 미공유 → 이미 쓰는 Redis로 TTL 이관이 다음 스텝 (Q9·P1)
  2. WebSocket은 CONNECT 1회 인증이라 소켓 유지 중 만료/로그아웃 미반영 → SEND 시 재검증 (P3)
  3. 계정 잠금에 해제 로직이 없고 취미 조회에 N+1이 있음 → 시간기반 잠금 / 조인 매핑 (P6·P7)

⛔ 절대 방어선 — 여기서 틀리면 치명적 3

  1. “Stateless인데 로그아웃?” — 블랙리스트로 강제 무효화, 트레이드오프까지 설명 (Q8·P1)
  2. “WebSocket 인증은 어디서?” — HTTP 필터 아님, STOMP CONNECT 채널 인터셉터 (Q15·P3)
  3. “Gateway 네가 만들었어?” — 팀 공용, 나는 Core 담당 + 필터 예외 설정 수정. 절대 과대포장 금지 (Q3)

🎤 1분 자기소개 스크립트 (검증된 사실만)

“Haru는 취미 공유·중고 거래·실시간 채팅을 결합한 지역기반 서비스이고, Gateway·CoreService·AssistService로 나뉜 마이크로서비스입니다. 저는 CoreService 백엔드에서 인증·채팅·게시판·취미 도메인을 맡았습니다. 인증은 Spring Security 무상태 + JWT로 구현하고 로그인 실패 잠금과 로그아웃 블랙리스트로 방어선을 뒀습니다. 가장 공들인 건 실시간 채팅인데, WebSocket 인증은 STOMP CONNECT 인터셉터에서 JWT로 처리하고, 메시지 확산은 Redis Pub/Sub로 발행하고 구독 리스너가 STOMP로 중계해 단일 서버에 묶이지 않게 했습니다. 기능을 넘어서 접근제어·트랜잭션 정합성을 챙기려 했고, 지금은 블랙리스트의 Redis 이관 같은 개선점을 정리해 두고 있습니다.”
