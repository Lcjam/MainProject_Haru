Haru — 구현 개요

취미 기반 매칭 · 위치기반 중고마켓 · 실시간 채팅 · AI 챗봇을 제공하는 마이크로서비스 웹 애플리케이션. 프론트엔드는 단일 게이트웨이만 바라보고, 게이트웨이가 JWT 인증과 라우팅을 담당한다.

Spring Boot 3.xSpring Cloud Gateway MyBatisRedisWebSocket / STOMP OAuth2 + JWTReact 19TypeScript Redux ToolkitLeafletOpenCV.js FastAPI · TinyLlamaMySQLDocker

아키텍처

Frontend Gateway CoreService AssistService FastAPI(AI) Data
요청 흐름: 프론트(3000) → GateWay(8080) → Core(8081) / Assist(8082) / FastAPI(8001). WebSocket은 ws://localhost:8080/ws → Gateway → Core(STOMP). 프론트는 게이트웨이 외 다른 서비스를 직접 호출하지 않는다.

핵심 플로우

1. 인증 (JWT)

React → POST /api/core/auth/login → GateWay(화이트리스트, 우회) → Core Core: 비밀번호 검증 → 실패누적/계정잠금 체크 → JWT 발급 → 응답 이후 모든 요청: Authorization: Bearer <jwt> → GateWay JwtAuthenticationFilter 검증 → 통과 시 라우팅

2. 실시간 채팅 (WebSocket / STOMP)

React @stomp/stompjs → CONNECT ws://localhost:8080/ws (헤더에 JWT) GateWay → Core WebSocketConfig: CONNECT 시 JWT 검증 → Principal 설정 전송: SEND /app/chat/send → Core → 저장(messages) → /topic/chatroom/{id} 브로드캐스트 알림: Redis Pub/Sub → /queue/notifications/{userEmail}

3. 마켓 거래

상품등록(이미지 멀티파트) → 근처상품 조회(거리기반) → 거래요청 with-chat → 채팅방+알림 자동 생성 → 판매자 승인 → 거래/결제 생성

4. AI 챗봇

React → /api/assist/tinylamanaver/chat → Assist Assist: chat_messages 히스토리 조회 → FastAPI(:8001 TinyLlama) 호출 → 번역 → 응답 저장

대표 구현 (STAR 요약)

상세 서술은 docs/portfolio/의 STAR·PAAR·Interview QA 마크다운 참조.

로그인 — 무차별 대입 방어

S: 비밀번호 공격 노출. A: 실패횟수 누적 + 임계치 계정잠금 + JWT 발급/블랙리스트. R: 반복 실패 차단·상태 기반 로그인 제어.

취미 — 도메인 검증

S: 잘못된 카테고리-취미 조합 유입. A: category_hobbies 관계 기반 서버 검증. R: 데이터 정합성 보장.

보드 — 역할 기반 접근제어

S: 권한 없는 보드 조작. A: 서비스 계층에서 멤버/방장 권한 검증(컨트롤러만 아님). R: 우회 불가한 인가.

채팅 — REST/WS 책임 분리

S: 실시간성과 신뢰성 동시 요구. A: 전송은 STOMP, 히스토리·읽음은 REST. R: 역할 분리로 단순·견고.

주요 Before → After

🔐 시크릿 외부화

- jwt.secret=teamsketchprojectsecretkey... + jwt.secret=${JWT_SECRET:haru-local-dev-...}

JWT secret·OAuth2 자격증명·redirect·CORS 출처·게이트웨이 호스트·번역 API 키를 모두 환경변수로. 로컬은 dev 기본값으로 무중단 동작, prod는 주입.

🐛 로컬 챗봇 버그 수정

- activeProfile = "prod" // 하드코딩 - → 로컬에서도 gateway-container 호출 + @Value("${gateway.base-url:localhost:8080}")

프로필이 항상 "prod"로 고정돼 로컬에서 컨테이너 호스트를 부르던 버그 제거.

📋 에러 응답 표준화

검증 실패 → 401 + 깨진 한글(???) + @RestControllerAdvice → 400 + UTF-8 JSON

@Valid 실패가 security entry point로 흘러 401로 나오던 문제를, 전역 핸들러로 일관된 400 ApiResponse로 교정.

🧪 테스트 도입

- 테스트 0개 + 인증 DTO 검증 6 cases (JUnit5) + build.gradle useJUnitPlatform()

JUnit Platform 미설정으로 테스트가 발견조차 안 되던 것까지 해결. DB 불필요한 순수 단위 테스트.

🧩 거대 컴포넌트 분리

- ChatRoom.tsx 741줄 (UI+상태+이펙트+검증 혼재) + ChatApprovalFlow · ChatMessageList · ChatInputZone 추출 → 601줄 + CameraModal 401→108: 카메라/OpenCV 로직 useCardScanner 훅으로

기능 보존하며 4개 거대 컴포넌트를 프레젠테이셔널/훅/순수함수로 분리. App의 인라인 NotificationHandler는 매 렌더 재생성 버그까지 해소. validateProductForm은 단위테스트 8개 동반. 실브라우저(Claude-in-Chrome)로 채팅·등록폼·카메라 모달 동작 검증.

🧹 컨트롤러 보일러플레이트 제거

- BoardController 13개 메서드 × (401 가드 + try/catch 3블록) + tokenUtils.requireEmail(token) + 서비스 위임만 → 426줄→244줄

email==null→401·IllegalArgumentException→400·Exception→500 세 패턴이 GlobalExceptionHandler와 정확히 일치 → 전역 핸들러로 위임. 401만 미커버였어서 UnauthorizedException+핸들러를 추가해 응답 형태(ApiResponse status·code·data) 보존. 계약 테스트 4개(401/400위임/200/인라인400)로 고정. 부수효과로 createBoard가 모든 예외를 400으로 숨기던 잠재 버그도 정규화(500).

📦 파일 I/O → 서비스 이관

- ProductController.getProductImage: 경로해석·존재확인·MIME감지 + ProductService.getProductImageResource() → 컨트롤러는 HTTP 헤더 조립만

Fat 컨트롤러의 비즈니스 로직을 서비스로 이관(ProductImageMapper 의존성 제거). 반환은 중첩 레코드 ProductImageFile. 계약 테스트 2개(존재 200+헤더 / 미존재 404)로 고정.

🤖 AI 챗봇 로컬 E2E 복구

- 게이트웨이 /api/fastapi/** 에 JWT 필터 → 토큰 없는 서버콜백 항상 401 - 빈 자격증명으로 번역 API 403 → 응답이 "INVALID EMAIL PROVIDED" + FastAPI 라우트 JWT 제외 · 빈 key/email 파라미터 생략 + transformers/torch → llama.cpp(gguf), 모델경로 env 외부화

사용자 진입점 /api/assist/**는 JWT 보호를 유지한 채, 유일한 호출자가 사용자 토큰을 안 싣는 서버-투-서버 콜백인 /api/fastapi/**만 필터 제외(WS 라우트 선례). 번역 단계의 빈 자격증명 403도 수정. 모델은 transformers+torch→llama.cpp(gguf, Apple Silicon Metal)로 바꾸고 윈도우 절대경로 하드코딩을 LLAMA_MODEL_PATH env로 외부화. 로그인 토큰으로 게이트웨이→Assist→번역→FastAPI(TinyLlama)→번역 전 구간 한국어 왕복 E2E 검증.

트러블슈팅 사례 — 게이트웨이 / Core 인가(authorization) 화이트리스트 불일치

실 사용 중 발견한 "간헐적 버그"를 라이브 디버깅으로 근본 원인까지 추적하고, 여러 해결안의 트레이드오프를 비교해 최소·정합 수정으로 끝낸 사례.

🐞 증상 (Problem)

회원가입·상품등록 화면에서 카테고리/취미 드롭다운이 떴다 안 떴다 했다. DB에는 카테고리 20종·취미 데이터가 분명히 있고, 쿼리·프론트 파싱도 정상인데 화면만 비었다. 재현이 들쭉날쭉해 원인 파악이 까다로웠다.

🔬 원인 진단 (Root Cause)

브라우저 확장 기반 라이브 디버깅(네트워크·콘솔 + 페이지 컨텍스트 fetch)으로 호출을 직접 관찰했다. 같은 페이지에서 토큰 유무만 바꿔 호출하니 결정적 단서가 나왔다:

GET /api/core/hobbies/categories (토큰 없음) → 401 "로그인이 필요한 서비스입니다." GET /api/core/hobbies/categories (유효 토큰) → 200 + data[20]

CoreService SecurityConfig는 이 경로를 이미 permitAll로 공개했지만, 게이트웨이 JwtAuthenticationFilter.PUBLIC_PATHS에 누락돼 있었다. 즉 게이트웨이가 Core에 닿기도 전에 401로 끊은 것 — 다계층 인가의 화이트리스트 불일치. 그래서 토큰이 유효할 때만 통과했고, 회원가입은 항상 비고(로그아웃), 상품등록은 토큰 만료 시 재현되어 "간헐적"으로 보였다.

⚖️ 해결 옵션 비교 (Options & Trade-offs)

옵션장점단점
① 프론트에 카테고리/취미 정적 하드코딩네트워크 호출 제거, 즉시 표시DB와 이중 관리, 데이터 변경 때마다 재배포, 단일 출처(SSOT) 위배
② 공개 화면용 임시 토큰 발급 후 호출기존 인증 경로 재사용공개 참조 데이터에 인증을 끼우는 부자연스러움, 인증 흐름·발급 로직 복잡도↑
③ 게이트웨이 PUBLIC_PATHS에 hobby 경로 추가 → Core와 정합 (채택)1파일·2줄 수정, DB 단일 출처 유지, 인증 의존 제거로 토큰 만료와 무관하게 동작게이트웨이/Core 두 곳의 화이트리스트를 함께 관리해야 함(정합성 책임)

✅ 선택 & 근거 (Decision)

취미·카테고리는 본질적으로 공개 참조 데이터이고, Core가 이미 공개로 설계해 둔 상태였다. 따라서 "게이트웨이를 Core 정책에 맞춰 정합"시키는 ③이 의도에 가장 부합하고, 변경 범위가 가장 작으며(=회귀 위험 최소), 데이터 단일 출처도 지킨다.

// GateWay JwtAuthenticationFilter.PUBLIC_PATHS — 누락 + "/api/core/hobbies", + "/api/core/hobbies/**", // CoreService SecurityConfig와 동일하게 공개

검증: 토큰 없이 두 엔드포인트 200 회복 · 로그아웃 상태 회원가입에서 드롭다운 21개 옵션 정상 렌더 · 보호 라우트(상품 등록)는 여전히 401 → 보안 회귀 없음.

💡 배운 점: 게이트웨이 + 서비스로 인가가 이중화된 구조에서는 공개 경로 화이트리스트가 단일 출처에서 어긋나기 쉽다. "DB·서비스는 맞는데 화면만 빈다"면 가장 바깥 계층(게이트웨이)의 인가부터 의심하고, 계층별 화이트리스트 정합을 점검해야 한다.

토큰 추출 단일화 ✅

substring(7)/.replace("Bearer ","")로 흩어진 Bearer 제거를 상위 호출부(ProfileController·AuthService·WebSocketConfig·Assist LlamaController)만 TokenUtils.extractTokenWithoutBearer로 통합. WebSocketConfig는 전역 .replace→접두사 전용 strip(실 JWT엔 동치, STOMP 핸드셰이크 보존). 보안 프리미티브(JwtTokenProvider·필터)·Gateway는 순환의존/서비스 분리로 제외. 계약 단위 테스트 동반(Core 12→16, Assist 첫 테스트 6).

응답 래퍼 — 도메인 경계 문서화 ✅

실측 결과 BaseResponse(7파일)/ApiResponse(15파일)는 한 파일 혼재가 아니라 도메인별로 갈린 평행 래퍼(JSON 계약 자체가 다름). 풀 단일화는 프론트 호출부까지 동시 변경되는 高위험이라, 경계 명문화 + "신규=ApiResponse+@RestControllerAdvice" 규약으로 마감.

FastAPI AI 챗봇 로컬 E2E 통합 ✅

llamaServer.py를 transformers+torch→llama.cpp(gguf)로 전환(Apple Silicon Metal, 모델 경로 LLAMA_MODEL_PATH 외부화·윈도우 절대경로 하드코딩 제거). 체인 두 버그 수정: ①게이트웨이 /api/fastapi/**의 JWT 필터가 토큰 없는 서버-투-서버 콜백을 401로 끊던 문제 ②빈 자격증명으로 MyMemory 번역이 403 INVALID EMAIL을 반환하던 문제. 로그인 토큰 → GW /api/assist/tinylamanaver/chat 한국어 질문 → 한국어 답변(번역 ko→en→TinyLlama→en→ko) E2E 검증·chat_messages 영속 확인.
